### 2. 與既有 ADR 無衝突

| 對照 ADR | 檢查重點 | 結果 |
|----------|----------|------|
| **ADR-003**（grype DB 離線快照） | ADR-012 是否延續離線 env 集合、是否與 DB 更新 SOP 衝突 | ✅ 無衝突。ADR-012 決策 2「烤死離線 env」完整複用 ADR-003 定義的 `GRYPE_DB_CACHE_DIR`、`GRYPE_DB_AUTO_UPDATE=false`、`GRYPE_DB_VALIDATE_AGE=false`，並額外收斂 `SYFT_CHECK_FOR_APP_UPDATE`/`GRYPE_CHECK_FOR_APP_UPDATE`（明確定性為「現行裸機 wrapper 未關的 outbound 洩漏點」，屬於**向後回補既有缺口**而非新增衝突）。DB 更新機制（決策 4：選項 B，volume 掛載）明示「直接沿用 DELIVERY_SOP 既有 DB 更新 SOP」「不需更新 image」，與 ADR-003 的月更 SOP 完全相容、無重複定義。 |
| **ADR-007**（交付封裝與 minisign 信任錨） | 容器交付是否繞過或另立信任錨；是否明示裸機路線不變 | ✅ 無衝突。ADR-012 決策 6 明確「同一把公鑰驗所有交付物」，簽章標的改為 `docker save` tar（而非 image 本身走 cosign/registry 簽章），與 ADR-007 的 minisign + SHA256SUMS 信任錨模式**同構延伸**，非另立新機制。Relations 段落明示「沿用：ADR-007（minisign 信任錨；裸機包路線不變）」。背景段落也明示「不取代 ADR-007/010 的裸機離線包（CLI 模式）」。 |
| **ADR-010**（雙平台 release；release.yml 職責） | docker.yml 是否與 release.yml 職責重疊或搶奪權限 | ✅ 無衝突。決策 5 明示新增**獨立檔案** `.github/workflows/docker.yml`，並特別註明「獨立於 release.yml——權限最小化 `packages: write` 各自宣告」，Relations 段落確認「release.yml 職責不變；docker.yml 獨立」。經查現況 `.github/workflows/` 目前僅有 `ci.yml`、`release.yml`，無任何 docker/container/ghcr 相關 workflow，確認新增 `docker.yml` 不會與既有職責衝突。 |

**結論**：三份既有 ADR 的邊界、SOP、信任錨均被**明示延伸而非覆寫**，「容器是新增交付形態、不取代裸機包」的定位在背景段落與 Relations 段落**雙重重申**，衝突風險低。

### 3. 鐵則相容性（`/home/ubuntu/CyTrace/CLAUDE.md`）

| 鐵則 | 檢查結果 |
|------|----------|
| **零外連（air-gapped）** | ✅ 通過。決策 2 完整列出離線 env 集合，且主動識別並回補現行裸機 wrapper 的 update-check outbound 洩漏點（`SYFT_CHECK_FOR_APP_UPDATE`/`GRYPE_CHECK_FOR_APP_UPDATE`），列入「後續追蹤」確保回補落地。決策 3 healthcheck 選擇「新子命令 `cytrace health`（TCP connect 檢查 bind port）」而非依賴 shell/curl，理由含「distroless 無 shell、避開 TLS 分歧、零新依賴」，避免引入額外 outbound 或攻擊面。後果段落誠實揭露「零外連在容器內靠 env + 無 client 依賴；**網路層隔離仍是場域責任**」，未過度承諾，此為恰當的責任邊界劃分。Build 期（S1 frontend-builder pnpm install、S2 rust cargo build、S3 engines 下載官方 release tar.gz）屬 CI 期網路存取，依 CLAUDE.md「零外連」鐵則僅限**執行期**，build 期 CI 有網屬允許範圍，ADR 未誤將兩者混淆。 |
| **供應鏈純淨** | ✅ 通過。S1/S2/S4 base image 均以 `@sha256:<digest>` 釘選（node:22-alpine、rust:1.95.0-slim-bookworm、gcr.io/distroless/static-debian12:nonroot）；S3 engines 下載後「SHA256 比對不符即 build fail（checksum 硬釘進 versions.env，可稽核）」，滿足「引擎 SHA256 驗證」要求。決策 4 選擇 distroless 而非 alpine，理由包含「busybox/musl CVE 噪音」考量，呼應純淨供應鏈精神。**待補注意**：ADR 未提及 NOTICE 檔案是否需因新增 base image（node:22-alpine、rust:1.95.0-slim-bookworm、distroless）而更新第三方授權標示——經查專案根目錄目前無 NOTICE 檔案存在（僅 ADR-007 提及「NOTICE（第三方授權）」為交付內容之一），此為既有缺口非本 ADR 新增，但容器化會使 base image 授權追蹤面擴大，建議在後續 FIRM 階段或實作 PR 中一併確認 NOTICE 涵蓋範圍，不影響本次 Draft 品質判定。未見任何中國來源依賴引入（node/rust/distroless 均為主流國際來源），符合「明確禁止中國來源依賴」要求。 |
| **破壞性操作防護（docker push）** | ✅ 通過。決策 8 明示「本機 Make targets：`docker-build`/`docker-smoke`/`docker-save`（**不含 push**——`.claude/settings.json` deny 維持；push 只由 CI 執行）」。經查 `.claude/settings.json` 現況，`permissions.deny` 已含 `"Bash(docker push *)"` 與 `"Bash(docker deploy *)"`，與 CLAUDE.md 鐵則表列「`docker push` 必須人類確認」完全對齊，ADR 決策未與此鐵則牴觸，且明確不在本機提供 push 能力，只由 CI workflow（決策 5：冒煙通過才 push）代為執行，符合最小權限原則。 |

**結論**：三項鐵則全數相容，無違反情形。供應鏈純淨項有一個**非阻斷性**的待補提醒（NOTICE 涵蓋範圍），不構成 FAIL 或 CONDITIONAL PASS 理由。
