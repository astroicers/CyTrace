## 背景（Context）

需求明定**強制使用 i18n 處理語言**，且**只支援繁體中文（台灣 zh-TW）與英文（美國 en-US）**。
此規則同時涵蓋**前端報表**與**CLI 輸出**兩個使用者可見面。硬編碼字串將使後續維護與審查失控，
也違反 `frontend_quality` profile 的把關。

## 評估選項（Options Considered）

### 選項 A：前端 react-i18next + CLI 共用 locale 鍵（單一詞彙來源）
- **優點**：前後端共用同一組訊息鍵與語系資源（`locales/zh-TW`、`locales/en-US`）；`frontend_quality` 可自動掃硬編碼；切換語言一致。
- **缺點**：需在 Rust CLI 端實作輕量 catalog 載入（讀同格式 JSON）。
- **風險**：鍵漂移 → 以 CI 檢查「兩語系鍵集合一致 + 無缺鍵」緩解。

### 選項 B：前端與 CLI 各自獨立 i18n
- **優點**：實作各自簡單。
- **缺點**：詞彙不一致、雙重維護、易漏譯。
- **風險**：審查時術語不一致。

## 決策（Decision）

採用 **選項 A**：**`react-i18next`** 驅動前端，`zh-TW` 為 **fallback**、`en-US` 並列；
語系資源置於共用 `locales/{zh-TW,en-US}.json`（命名空間分區）。**Rust CLI 載入相同 JSON 鍵**輸出訊息。
**禁止硬編碼任何使用者可見字串**；嚴重度標籤、報表區塊標題、CLI 提示一律走鍵。

**共用值契約（不只鍵一致，值語意也要一致）：**
- 釘一種插值語法 **`{{var}}`**，Rust loader 實作同樣的簡單替換。
- **共用命名空間禁用** 複數/context/nesting（i18next 的 `_one`/`_other`、`_male`、`$t()`）；CLI-facing 字串僅用 `{{var}}`。
  否則前端寫 `{{count}}`/複數族，在 CLI 會渲染成字面值或缺變體——鍵同步但值默默發散。

**CI 檢查（NFR-06）：**
- 做**遞迴葉鍵集合 diff**（symmetric difference）涵蓋兩語系 + Rust catalog，因 locale 為巢狀命名空間。
- 同時 lint：共用值只能用 Rust loader 支援的特性。
- ⚠️ ASP 內建 `make i18n-check` 只比**頂層鍵數量**，不足以守巢狀「無缺鍵」；T003 須自建遞迴檢查，不可把 `make i18n-check` 當作滿足 NFR-06。

## 後果（Consequences）

**正面影響：**
- 全產品雙語一致、可審查；新增字串強制走鍵，杜絕硬編碼。

**負面影響 / 技術債：**
- 需維護鍵集合一致性檢查；Rust 端需一個輕量 catalog 載入器。

**後續追蹤：**
- 嚴重度標籤鍵與 ADR-006 對齊；報表區塊鍵與 UIUX_SPEC 對齊。
