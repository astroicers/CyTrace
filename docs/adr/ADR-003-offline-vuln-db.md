## 背景（Context）

Grype 比對 CVE 需要漏洞資料庫（grype DB）。預設行為是線上 `grype db update` 取最新庫，
但 CyTrace 場域**無網際網路**，且進場更新申請耗時。必須在**完全離線**下仍能完整比對，
並提供可控、可稽核的 DB 更新途徑。

## 評估選項（Options Considered）

### 選項 A：安裝包預載 grype DB 快照 + 離線匯入更新 SOP
- **優點**：離線即可完整比對；DB 版本明確、可記錄於報表（資料時效可稽核）；更新走人工攜入快照，符合機房流程。
- **缺點**：DB 時效取決於攜入頻率（資料新鮮度受流程限制）。
- **風險**：DB 過舊導致漏報 → 以報表標註 DB 快照日期 + 更新 SOP 緩解。

### 選項 B：要求現場連線更新
- **優點**：DB 最新。
- **缺點**：違反 air-gapped 前提，不可行。

## 決策（Decision）

採用 **選項 A**：**安裝包預載 grype DB 離線快照**，強制離線並指向本地 DB 路徑。提供**離線更新 SOP**：
在有網段以 `grype db update` 取得新庫 → 打包快照 → 攜入目標機匯入。**報表須顯示 DB 快照版本與建立日期**，使資料時效可被稽核。

**離線環境變數（完整集合，缺一不可）：**

| 變數 | 值 | 原因 |
|------|----|------|
| `GRYPE_DB_AUTO_UPDATE` | `false` | 關閉線上自動更新 |
| `GRYPE_DB_VALIDATE_AGE` | `false` | **關鍵**：Grype 預設會驗證 DB 年齡（`max-allowed-built-age` 預設 120h＝5 天），DB 過舊即**中止掃描**。軍用快照本就常是數週/數月舊，不關掉此驗證則第一次 `cytrace run` 就會以「database too old」失敗。 |
| `GRYPE_DB_CACHE_DIR` / `--db` | 指向安裝包內快照路徑 | 不依賴 PATH 上任意 DB |

> 關閉年齡驗證是**刻意的 air-gapped 取捨**：時效風險改由報表顯示快照日期（ADR-005 / UIUX_SPEC）＋ 離線更新 SOP 控管，而非靠 Grype 的年齡守門。
> 若偏好有上限的守門，可改設 `GRYPE_DB_MAX_ALLOWED_BUILT_AGE` 為刻意的大值（如 `8760h`＝1 年）。

## 後果（Consequences）

**正面影響：**
- 完全離線可用；資料時效透明、可稽核；更新流程與機房作業相容。

**負面影響 / 技術債：**
- 需維護「離線 DB 快照打包/匯入」工具與 SOP 文件。

**後續追蹤：**
- 封裝細節見 ADR-007；報表時效標註見 ADR-005 / UIUX_SPEC。

## 成功指標（Success Metrics）

| 指標 | 目標值 | 驗證方式 | 檢查時間 |
|------|--------|----------|----------|
| 離線完整比對 | 通過 | 斷網環境跑 `cytrace scan` 並產出漏洞 | M1 |
| 舊 DB 快照不被年齡驗證中止 | 通過 | 以刻意回溯 30 天的 DB 快照跑 `cytrace run`，不出現 age-validation 中止 | M1 |
| 報表顯示 DB 版本/日期 | 是 | 檢視報表元資料區 | M3 |
| 更新 SOP 可重現 | 是 | 依 SOP 由快照匯入並驗證 DB 生效 | M4 |
